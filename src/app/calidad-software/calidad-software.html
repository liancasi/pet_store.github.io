Atributos de Calidad del Software (ISO/IEC 25010)

Funcionalidad

  • ·Completitud: el sistema cubre todos los requisitos funcionales definidos en el ERS.
  • ·Correctitud: las funciones producen resultados exactos ante entradas validas.
  • ·Apropiabilidad: las funciones satisfacen las tareas clínicas y administrativas del cliente.

Rendimiento

  • ·Tiempo de respuesta: API menor a 300ms para operaciones CRUD basicas.
  • ·Utilizacion de recursos: consumo de CPU y memoria dentro de limites aceptables.
  • ·Capacidad: soporte concurrente de hasta 50 usuarios simultáneos.

Seguridad

  • ·Confidencialidad: datos clinicos cifrados, acceso solo a usuarios autorizados (JWT).
  • ·Integridad: validación de entradas, control de concurrencia y auditoría de cambios.
  • ·No repudio: registro de acciones criticas con usuario y marca de tiempo.

Mantenibilidad

  • ·Modularidad: arquitectura de capas con separación de responsabilidades (Clean Architecture).
  • ·Analizabilidad: código legible y con complejidad ciclomática máxima de 10.
  • ·Testabilidad: cobertura de pruebas mínima del 80% medida con JaCoCo.

Estimación mediante Puntos Función

¿Qué son los Puntos de Función?

Los Puntos de Función (IFPUG) son una técnica estandarizada (ISO/IEC 20926) para medir el tamaño funcional de un sistema desde la perspectiva del usuario, independientemente del lenguaje o tecnología usada. Se identifican cinco tipos de componentes: EI — formularios que el usuario envía; EO — reportes o resultados calculados que el sistema devuelve; EQ — búsquedas simples sin transformación de datos; ILF — tablas principales mantenidas por el sistema; EIF — datos de sistemas externos consumidos. Cada componente se pondera según su complejidad (baja / media / alta) y la suma da los Puntos de Función No Ajustados (PFNA). Con ese número se puede estimar esfuerzo, costo y duración del proyecto antes de escribir una sola línea de código.

Casos de Uso 2.0

¿Qué es Casos de Uso 2.0?

Propuesta por Ivar Jacobson (2011) como evolución de los Casos de Uso clásicos de UML, Casos de Uso 2.0 organiza la funcionalidad del sistema en porciones más pequeñas y accionables llamadas rebanadas de caso de uso (use case slices). Cada rebanada describe una parte concreta del caso de uso que entrega valor real y puede ser implementada, probada y desplegada de forma independiente. Esto facilita la planificación iterativa e incremental: en lugar de especificar todo antes de desarrollar, el equipo trabaja rebanada a rebanada, entregando valor al cliente en cada iteración y ajustando los requisitos según la retroalimentación recibida.

Se aplica la técnica de Casos de Uso 2.0 (Ivar Jacobson), que organiza la funcionalidad en rebanadas (use case slices) para la entrega incremental de valor.

Modelo 4+1 de Kruchten Aplicado a PetStore

¿Qué es el Modelo 4+1?

Propuesto por Philippe Kruchten en 1995, el Modelo 4+1 describe la arquitectura de un sistema de software desde cinco perspectivas complementarias, cada una orientada a un stakeholder diferente. Las cuatro vistas principales son: Vista Lógica — estructura de clases y componentes para el desarrollador; Vista de Proceso — flujo de ejecución y concurrencia para analistas de rendimiento; Vista de Implementación — organización del código fuente para el equipo de desarrollo; Vista Física — despliegue en infraestructura para operaciones. La quinta vista, Escenarios (+1), usa casos de uso para ilustrar y validar cómo interactúan las cuatro anteriores. Juntas ofrecen una descripción arquitectónica completa que puede comunicarse a todos los involucrados del proyecto sin importar su perfil técnico.

Vista Lógica

Diagramas de clases y secuencia que modelan los módulos: Patient, Appointment, Invoice, Inventory, User.

Vista de Proceso

Diagrama de actividades que muestra el flujo de una consulta: ingreso del paciente, atención, prescripcion y facturación.

Vista de Desarrollo

Estructura de paquetes: controller / service / repository / dto / config. Modelo de tres capas (Presentación, Negocio, Datos).

Vista Fisica

Servidor cloud: Backend Spring Boot en Docker, base de datos PostgreSQL administrada, frontend React en CDN/hosting estático.

Vista de Escenarios (+1)

Casos de uso prioritarios: Registrar mascota, Agendar cita, Registrar consulta, Generar factura — que guían y validan las cuatro vistas anteriores.

Errores Comunes y Optimización del Análisis y Diseño

¿Por qué importa detectarlos temprano?

En ingeniería de software, los errores cometidos en las fases de análisis y diseño son los más costosos de corregir: un requisito mal especificado descubierto en producción puede costar hasta 200 veces más que si se hubiera detectado en la fase inicial (Boehm, 1981). Los errores más frecuentes incluyen: requisitos ambiguos sin criterios de aceptación; clases con múltiples responsabilidades (violación del SRP); acoplamiento alto entre capas que impide la reutilización; diseño "big bang" sin iteraciones ni validación temprana; y ausencia de trazabilidad entre requisitos y código. La optimización del análisis y diseño implica aplicar revisiones formales (walkthroughs), prototipado rápido, pruebas de arquitectura y retroalimentación continua del cliente para detectar y corregir estos errores antes de que se conviertan en deuda técnica irreversible.

Errores Comunes

  • –Requisitos ambiguos sin criterios de aceptación
  • –Clases con multiples responsabilidades (violacion SRP)
  • –Acoplamiento alto entre capas de la aplicación
  • –Omision de casos de uso alternativos y de error
  • –Diagramas UML no actualizados con el código real

Optimizacion Aplicada

  • +Revisión formal de ERS con stakeholders antes de cada fase
  • +Aplicación de patrones de diseño (Repository, DTO, Factory)
  • +Arquitectura de capas con inyeccion de dependencias Spring
  • +Definition of Done con criterios de aceptación por cada US
  • +Documentación API automática con Swagger / OpenAPI

Principios SOLID y DRY aplicados a PetStore

¿Qué son SOLID y DRY?

SOLID es un acrónimo propuesto por Robert C. Martin que reúne cinco principios de diseño orientado a objetos para crear software mantenible y escalable: S (Single Responsibility) — una clase, una sola razón para cambiar; O (Open/Closed) — abierto a extensión, cerrado a modificación; L (Liskov Substitution) — las subclases deben poder reemplazar a sus superclases sin alterar el comportamiento; I (Interface Segregation) — interfaces específicas en lugar de una genérica; D (Dependency Inversion) — depender de abstracciones, no de implementaciones concretas. DRY (Don't Repeat Yourself) complementa SOLID estableciendo que cada pieza de conocimiento debe tener una única representación en el sistema — la duplicación de lógica es la raíz de la mayoría de los errores de mantenimiento. Aplicar SOLID + DRY reduce el acoplamiento, aumenta la cohesión y hace que el código sea testeable de forma unitaria.

Reglas de Código Limpio en la Implementación

¿Qué es Código Limpio?

El libro Clean Code: A Handbook of Agile Software Craftsmanship (Robert C. Martin, 2008) sistematiza las prácticas que distinguen el código profesional del código que "simplemente funciona". Sus principios centrales son: nombres expresivos que revelan intención sin necesidad de comentarios; funciones pequeñas con un único propósito (máximo ~20 líneas); comentarios solo para el "por qué" no obvio — nunca para explicar qué hace el código; manejo explícito de errores sin excepciones silenciosas; y estructura en capas con responsabilidades claras. La métrica principal para medir la limpieza es la Complejidad Ciclomática de McCabe (CC): un valor ≤ 10 por método garantiza que el código es testeable, legible y mantenible. El código limpio reduce la deuda técnica, facilita el onboarding de nuevos desarrolladores y es prerequisito para la sostenibilidad del proyecto a largo plazo.

Funciones

  • ·Máximo 20 líneas por método
  • ·Un solo nivel de abstracción
  • ·Nombres tipo verbo-sustantivo
  • ·Máximo 3 parámetros por función

Variables y Comentarios

  • ·Nombres que revelen la intención
  • ·Sin abreviaturas crípticas (petId no pid)
  • ·Comentarios solo cuando el WHY no es obvio
  • ·Sin código comentado en el repositorio

Manejo de Errores

  • ·Excepciones propias por dominio
  • ·@ControllerAdvice para errores HTTP
  • ·Nunca retornar null; usar Optional
  • ·Mensajes de error informativos

Clases y Concurrencia

  • ·Clases pequenas con responsabilidad única
  • ·@Transactional para operaciones criticas
  • ·Locks optimistas para control de concurrencia
  • ·Encapsulamiento correcto de atributos

Índice de McCabe (Complejidad Ciclomática)

El código limpio reduce directamente el índice de McCabe al eliminar ramas innecesarias. PetStore mantiene CC igual o menor a 10 por método.

  • ·Extracción de métodos reemplaza bloques if anidados
  • ·Polimorfismo sustituye cadenas de switch/case
  • ·Guard clauses reducen la profundidad del código

Resultados de Pruebas Unitarias (JUnit 5)

¿Qué son las pruebas unitarias con JUnit 5?

JUnit 5 es el framework estándar de pruebas unitarias para Java, compuesto por tres módulos: JUnit Platform (motor de ejecución en la JVM), JUnit Jupiter (nueva API con anotaciones como @Test, @BeforeEach, @ParameterizedTest) y JUnit Vintage (compatibilidad con JUnit 3/4). Una prueba unitaria verifica el comportamiento de una única unidad de código (método o clase) de forma aislada, usando Mockito para simular dependencias externas (repositorios, servicios, BD) sin necesidad de infraestructura real. El ciclo que seguimos es TDD — Red → Green → Refactor: primero se escribe la prueba que falla (Red), luego el código mínimo que la pasa (Green) y finalmente se limpia sin romper las pruebas (Refactor). JaCoCo mide la cobertura de ramas e instrucciones alcanzadas por las pruebas, generando el reporte HTML integrado en el pipeline de GitHub Actions.

Arquitectura Limpia (Robert C. Martin) en PetStore

PetStore adopta los principios de Arquitectura Limpia (Clean Architecture) de Robert C. Martin. Las dependencias siempre apuntan hacia adentro: desde la infraestructura hacia las reglas del negocio. El núcleo del sistema es independiente del framework, la base de datos y la interfaz de usuario.

Independencia y Limites en PetStore

Independencia del Framework

Las entidades y servicios de dominio no importan clases de Spring Boot. La lógica de negocio puede ejecutarse en cualquier entorno Java sin el framework.

Independencia de la Base de Datos

Las entidades de dominio no conocen JPA ni PostgreSQL. El repositorio es una interfaz en el dominio; la implementación JPA es un adaptador externo.

Independencia de la Interfaz

La capa de negocio no sabe que existe React. El frontend consume APIs REST; el backend no tiene referencias al cliente web.

Testabilidad

La independencia permite probar las reglas del negocio con JUnit sin levantar el contexto de Spring Boot, reduciendo el tiempo de ejecucion de pruebas.
