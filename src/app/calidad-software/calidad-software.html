Descripción General

El primer informe establece las bases conceptuales de la calidad de software aplicada al proyecto PetStore. Aborda los estándares internacionales, modelos de calidad y las métricas iniciales que guían el desarrollo de la plataforma veterinaria. Se identifican los atributos de calidad clave con base en el estándar ISO/IEC 25010.

ISO/IEC 25010 Modelos de Calidad Atributos de Calidad Métricas Iniciales Requisitos de Calidad
⭐
Atributos de Calidad Identificados

Calidad Funcional

  • Completitud funcional de los módulos
  • Exactitud en el registro de datos clínicos
  • Adecuación funcional para veterinarios
  • Interoperabilidad entre módulos

Calidad No Funcional

  • Fiabilidad del sistema en producción
  • Usabilidad para personal veterinario
  • Eficiencia en tiempos de respuesta
  • Mantenibilidad y portabilidad del código
Meta de cobertura QA: El proyecto establece cobertura de pruebas ≥ 80% y cero errores críticos en producción al momento del lanzamiento Beta.

Código Limpio: Principios, Prácticas y Métricas de Calidad

SOLID · DRY · Complejidad Ciclomática · Análisis Estático · Refactoring en PetStore

🧱
Principios SOLID y DRY en PetStore
PrincipioDescripciónAplicación en PetStore
S — SRPSingle Responsibility PrincipleCada servicio (PetService, LoginService) tiene una única responsabilidad bien definida
O — OCPOpen/Closed PrincipleLa arquitectura permite agregar módulos sin modificar los existentes
L — LSPLiskov Substitution PrincipleLas implementaciones de interfaces son intercambiables sin alterar el comportamiento
I — ISPInterface Segregation PrincipleInterfaces separadas por dominio (mascotas, usuarios, facturación)
D — DIPDependency Inversion PrincipleInyección de dependencias en Spring Boot para desacoplar capas
DRYDon't Repeat YourselfEliminación de código duplicado en validaciones y transformaciones de datos

Clases, Concurrencia y Refinamiento

Clases Limpias

  • Alta cohesión: atributos y métodos relacionados
  • Bajo acoplamiento entre capas (Controller → Service → Repository)
  • DTOs para separar capas de presentación y persistencia
  • Manejo de errores con @ControllerAdvice global

Concurrencia y Refinamiento

  • Transacciones (@Transactional) para consistencia de datos
  • Gestión segura de acceso concurrente a recursos
  • Refactoring disciplinado guiado por métricas SonarQube
  • Revisiones periódicas del código (code review)

Pruebas de Software

TDD · JUnit 5 · JaCoCo · Pruebas Unitarias, Integración, Sistema y Automatizadas

🧪
Tipos de Pruebas de Software

Pruebas Unitarias

  • Verifican unidades individuales (métodos, clases)
  • Aisladas con mocking de dependencias (Mockito)
  • Implementadas con JUnit 5 en Spring Boot
  • Ejecutadas automáticamente en pipeline CI

Pruebas de Integración

  • Interacción entre módulos del sistema
  • Comunicación entre capas Controller → Service → Repository
  • Pruebas de endpoints REST con datos reales
  • Base de datos H2 en memoria

Pruebas de Sistema y Aceptación

  • Validación end-to-end de flujos principales
  • UAT con la Veterinaria cliente
  • Validación de atributos no funcionales

Caja Blanca · Gris · Negra

  • Blanca: lógica interna, ramas y condiciones Java
  • Gris: endpoints REST + respuesta de base de datos
  • Negra: flujos de usuario en la interfaz web (UAT)
🔁
TDD – Desarrollo Guiado por Pruebas
Ciclo TDD en PetStore:

🔴 Red: Escribir la prueba que falla primero (define el comportamiento esperado)
🟢 Green: Implementar el código mínimo para que la prueba pase
♻️ Refactor: Mejorar el código manteniendo todas las pruebas en verde
