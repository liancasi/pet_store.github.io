Atributos de Calidad del Software (ISO/IEC 25010)

Funcionalidad

  • ·Completitud: el sistema cubre todos los requisitos funcionales definidos en el ERS.
  • ·Correctitud: las funciones producen resultados exactos ante entradas validas.
  • ·Apropiabilidad: las funciones satisfacen las tareas clinicas y administrativas del cliente.

Rendimiento

  • ·Tiempo de respuesta: API menor a 300ms para operaciones CRUD basicas.
  • ·Utilizacion de recursos: consumo de CPU y memoria dentro de limites aceptables.
  • ·Capacidad: soporte concurrente de hasta 50 usuarios simultaneos.

Seguridad

  • ·Confidencialidad: datos clinicos cifrados, acceso solo a usuarios autorizados (JWT).
  • ·Integridad: validacion de entradas, control de concurrencia y auditoria de cambios.
  • ·No repudio: registro de acciones criticas con usuario y marca de tiempo.

Mantenibilidad

  • ·Modularidad: arquitectura de capas con separacion de responsabilidades (Clean Architecture).
  • ·Analizabilidad: codigo legible y con complejidad ciclomatica maxima de 10.
  • ·Testabilidad: cobertura de pruebas minima del 80% medida con JaCoCo.

Gestion de Requisitos en PetStore

El proceso de gestion de requisitos sigue el estandar IEEE 830 / IEEE 29148 para la redaccion del ERS. Los requisitos se clasifican en funcionales y no funcionales, se priorizan usando MoSCoW y se gestionan con control de versiones.

Requisitos Funcionales (RF)

  • ·RF-01: Registro y autenticacion de usuarios por roles
  • ·RF-02: Registro de clientes y mascotas con historial
  • ·RF-03: Agendamiento de citas con notificaciones
  • ·RF-04: Registro de consultas y prescripciones
  • ·RF-05: Control de inventario con alertas de stock
  • ·RF-06: Facturacion y generacion de reportes
  • ·RF-07: Modulo de inteligencia artificial diagnostica

Requisitos No Funcionales (RNF)

  • ·RNF-01: Disponibilidad del sistema 99% del tiempo
  • ·RNF-02: Respuesta de API menor a 300ms
  • ·RNF-03: Autenticacion segura con JWT y BCrypt
  • ·RNF-04: Cobertura de pruebas minima del 80%
  • ·RNF-05: Interfaz responsive compatible con navegadores modernos
  • ·RNF-06: Datos clinicos bajo normativa de privacidad

Casos de Uso 2.0

Se aplica la tecnica de Casos de Uso 2.0 (Ivar Jacobson), que organiza la funcionalidad en rebanadas (use case slices) para la entrega incremental de valor.

UC1 Registrar Mascota

Actor: Recepcionista. Flujo: ingresar datos del propietario y la mascota, asignar ficha clinica, confirmar registro exitoso.

UC2 Gestionar Cita Medica

Actor: Recepcionista / Dueno. Flujo: seleccionar mascota, fecha y hora disponible, confirmar cita, enviar notificacion automatica.

UC3 Registrar Consulta y Prescripcion

Actor: Veterinario. Flujo: acceder a la ficha de la mascota, registrar sintomas, diagnostico, tratamiento y prescripciones medicas.

UC4 Generar Factura y Reporte

Actor: Recepcionista / Administrador. Flujo: seleccionar servicios prestados, calcular total, emitir factura PDF y registrar el pago.

Errores Comunes y Optimizacion del Analisis y Diseno

Errores Comunes

  • –Requisitos ambiguos sin criterios de aceptacion
  • –Clases con multiples responsabilidades (violacion SRP)
  • –Acoplamiento alto entre capas de la aplicacion
  • –Omision de casos de uso alternativos y de error
  • –Diagramas UML no actualizados con el codigo real

Optimizacion Aplicada

  • +Revision formal de ERS con stakeholders antes de cada fase
  • +Aplicacion de patrones de diseno (Repository, DTO, Factory)
  • +Arquitectura de capas con inyeccion de dependencias Spring
  • +Definition of Done con criterios de aceptacion por cada US
  • +Documentacion API automatica con Swagger / OpenAPI

Reglas de Codigo Limpio en la Implementacion

Funciones

  • ·Maximo 20 lineas por metodo
  • ·Un solo nivel de abstraccion
  • ·Nombres tipo verbo-sustantivo
  • ·Maximo 3 parametros por funcion

Variables y Comentarios

  • ·Nombres que revelen la intencion
  • ·Sin abreviaturas crípticas (petId no pid)
  • ·Comentarios solo cuando el WHY no es obvio
  • ·Sin codigo comentado en el repositorio

Manejo de Errores

  • ·Excepciones propias por dominio
  • ·@ControllerAdvice para errores HTTP
  • ·Nunca retornar null; usar Optional
  • ·Mensajes de error informativos

Clases y Concurrencia

  • ·Clases pequenas con responsabilidad unica
  • ·@Transactional para operaciones criticas
  • ·Locks optimistas para control de concurrencia
  • ·Encapsulamiento correcto de atributos

Indice de McCabe (Complejidad Ciclomatica)

El codigo limpio reduce directamente el indice de McCabe al eliminar ramas innecesarias. PetStore mantiene CC igual o menor a 10 por metodo.

  • ·Extraccion de metodos reemplaza bloques if anidados
  • ·Polimorfismo sustituye cadenas de switch/case
  • ·Guard clauses reducen la profundidad del codigo

Tipos de Pruebas Aplicadas

Pruebas Unitarias

Validan el comportamiento de metodos y clases de forma aislada, usando mocks para dependencias externas.

  • ·Herramientas: JUnit 5 + Mockito + AssertJ
  • ·Patron AAA: Arrange, Act, Assert
  • ·Cobertura medida con JaCoCo en CI/CD

Pruebas de Integracion

Verifican la interaccion entre capas: controlador, servicio y repositorio con base de datos real.

  • ·Herramientas: Spring Boot Test, H2 in-memory
  • ·@SpringBootTest con contexto completo
  • ·@DataJpaTest para capa de repositorio

TDD — Test Driven Development

Metodologia adoptada para el desarrollo de los modulos criticos del sistema.

  • ·Ciclo: Red (prueba falla) → Green (pasa) → Refactor
  • ·Garantiza cobertura desde el primer commit
  • ·Aplicado en autenticacion JWT y logica de citas

Pruebas de Aceptacion (UAT)

Validan que el sistema cumple los requisitos del cliente en entorno real antes del lanzamiento Beta.

  • ·Escenarios basados en los Casos de Uso 2.0
  • ·Participacion del personal de la clinica veterinaria
  • ·Aprobacion formal mediante acta de aceptacion

Arquitectura Limpia (Robert C. Martin) en PetStore

PetStore adopta los principios de Arquitectura Limpia (Clean Architecture) de Robert C. Martin. Las dependencias siempre apuntan hacia adentro: desde la infraestructura hacia las reglas del negocio. El nucleo del sistema es independiente del framework, la base de datos y la interfaz de usuario.

Independencia y Limites en PetStore

Independencia del Framework

Las entidades y servicios de dominio no importan clases de Spring Boot. La logica de negocio puede ejecutarse en cualquier entorno Java sin el framework.

Independencia de la Base de Datos

Las entidades de dominio no conocen JPA ni MySQL. El repositorio es una interfaz en el dominio; la implementacion JPA es un adaptador externo.

Independencia de la Interfaz

La capa de negocio no sabe que existe React. El frontend consume APIs REST; el backend no tiene referencias al cliente web.

Testabilidad

La independencia permite probar las reglas del negocio con JUnit sin levantar el contexto de Spring Boot, reduciendo el tiempo de ejecucion de pruebas.

Politica, Reglas del Negocio y Arquitectura Limpia

En Clean Architecture, las reglas del negocio son la parte mas valiosa del sistema. Se dividen en reglas criticas del negocio (entidades) y casos de uso que automatizan procesos especificos del sistema.

Entidades — Reglas Criticas del Negocio

  • ·Una mascota pertenece a exactamente un propietario registrado.
  • ·Una cita no puede solaparse con otra del mismo veterinario.
  • ·El historial clinico es inmutable una vez registrado por el veterinario.
  • ·Un medicamento no puede dispensarse si su stock es cero.

Casos de Uso — Reglas de la Aplicacion

  • ·AgendarCitaUseCase: valida disponibilidad y envia notificacion automatica.
  • ·RegistrarConsultaUseCase: crea el registro clinico y actualiza el historial.
  • ·GenerarFacturaUseCase: calcula el total y emite el comprobante digital.
  • ·AlertarStockUseCase: notifica cuando el inventario cae por debajo del minimo.
